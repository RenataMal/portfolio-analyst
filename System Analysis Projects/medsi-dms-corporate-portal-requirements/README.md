# Разработка требований для портала корпоративных клиентов МЕДСИ

## О проекте
Индивидуальный проект по системному анализу и управлению требованиями для HR-портала корпоративных клиентов сети клиник **МЕДСИ**.

В рамках проекта был спроектирован портал для HR-отделов компаний, использующих программы ДМС для сотрудников. Решение ориентировано на автоматизацию отчетности, прозрачный мониторинг использования ДМС и снижение объема ручных запросов в МЕДСИ и страховые компании.

## Цель проекта
Подготовить детализированный набор требований для разработки портала, который позволит корпоративным клиентам:
- получать сводную статистику по использованию ДМС;
- отслеживать статусы обращений;
- самостоятельно формировать отчеты;
- анализировать ключевые показатели по программам ДМС;
- сократить зависимость от ручной коммуникации с МЕДСИ.

## Бизнес-контекст
**МЕДСИ** - одна из крупнейших федеральных сетей частных медицинских клиник в России. Значительная часть клиентской базы компании - корпоративные клиенты и застрахованные по программам ДМС, что делает задачу цифровизации отчетности и мониторинга особенно актуальной.

## Проблемы текущего процесса
В ходе анализа были выявлены ключевые проблемы:
- отсутствие единой цифровой системы для мониторинга и отчетности по ДМС;
- использование разрозненных каналов связи: e-mail, телефон, Excel-выгрузки;
- низкая прозрачность данных для HR-отделов;
- сложности в отслеживании статусов запросов на медицинские услуги;
- высокая нагрузка на сотрудников МЕДСИ при подготовке отчетности вручную.

## Предлагаемое решение
Разработка **портала для корпоративных клиентов**, интегрированного с внутренними системами МЕДСИ и системами страховых компаний.

Портал должен обеспечивать:
- доступ HR-отделов к агрегированной статистике по ДМС;
- мониторинг использования лимитов;
- анализ причин отказов;
- фильтрацию данных по периоду, подразделению, типу услуги и статусу обращения;
- выгрузку отчетов в XLSX / CSV;
- отслеживание статусов запросов в режиме, близком к реальному времени.

## Целевые результаты проекта
В работе были сформулированы SMART-цели, в том числе:
- подключить к порталу не менее 70% корпоративных клиентов в пилотных клиниках к концу IV квартала 2026 года;
- обеспечить самостоятельное формирование не менее 60% регулярных отчетов HR-отделами в течение 12 месяцев после запуска;
- обеспечить прохождение 80% запросов по ДМС через систему с возможностью отслеживания статуса в течение 6 месяцев после запуска;
- снизить количество спорных случаев по запросам на 30% в течение первого года работы портала.

## Что было сделано
- проведен анализ предметной области и текущих бизнес-процессов;
- собраны и структурированы требования от заказчика и стейкхолдеров;
- определены регуляторные ограничения;
- выделены основные проблемы и предложены пути решения;
- выбрана модель жизненного цикла разработки;
- сформированы функциональные требования с приоритизацией по **MoSCoW**;
- сформированы нефункциональные требования по модели **FURPS+**;
- спроектирован REST API для получения сводной статистики по обращениям ДМС;
- подготовлена спецификация в формате **OpenAPI**;
- разработана концептуальная модель данных и ER-диаграмма;
- построены BPMN- и UML-диаграммы для описания процессов и архитектурной логики решения.

## Функциональные требования
К ключевым функциональным требованиям относятся:
- просмотр HR-отделом сводной статистики по обращениям сотрудников по программам ДМС;
- выгрузка отчетов по использованию ДМС;
- управление ролями и правами доступа;
- поиск и фильтрация данных по обращениям.

Основные требования были приоритизированы как **Must Have**, а расширенные сценарии — как **Should Have / Could Have**.

## Нефункциональные требования
В проекте отдельно проработаны требования к:
- **Usability** — понятный интерфейс для HR-менеджеров;
- **Reliability** — высокая доступность и быстрое восстановление после сбоев;
- **Performance** — ограничения по времени отклика и выгрузке отчетов;
- **Supportability** — удобство сопровождения и диагностики;
- **Security** — защита данных в соответствии с ФЗ-152 и ФЗ-323.

## Ключевой сценарий
Центральный пользовательский сценарий проекта — **формирование HR-отчета по ДМС**.

Для него были спроектированы:
- UI-фильтры;
- структура запроса и ответа;
- логика обработки ошибок;
- REST-метод:
`POST /api/hr/dms-usage/search`

Метод возвращает сводные показатели, включая:
- количество обращений;
- количество одобренных обращений;
- количество отказов;
- суммарные выплаты;
- долю сотрудников, использовавших ДМС;
- среднюю стоимость обращения;
- процент использования лимита.

## Модель данных
В концептуальной модели были выделены основные сущности:
- `company`
- `department`
- `employee`
- `dms_contract`
- `dms_service_type`
- `dms_request`

Модель позволяет рассчитывать ключевые операционные KPI по ДМС и поддерживает аналитические сценарии HR-портала.
